# 深圳共享单车再平衡项目

本仓库用于开展深圳共享单车区域再平衡优化项目。项目重点是基于订单、站点或区域需求数据，分析不同区域在不同时段的车辆供需差异，并构建再平衡模型与可视化分析结果。

当前协作重点是让三名成员明确各自负责的模块、代码放置位置和交付内容。

## 仓库结构

```text
shenzhen-bike-rebalancing/
├── data/        # 数据文件与数据说明
├── docs/        # 项目文档、任务说明、方法说明
├── notebooks/   # 数据探索、模型实验、可视化分析
├── outputs/     # 图表、模型输出、实验结果
├── src/         # 可复用代码
├── tests/       # 测试代码
├── .gitignore
└── README.md
```

## 三人分工

### 1. 数据负责人

主要负责数据的整理、清洗和说明。

具体任务包括：

- 明确数据来源、字段含义和数据时间范围。
- 整理原始数据和清洗后的数据。
- 编写数据预处理脚本。
- 说明缺失值、异常值和重复数据的处理方式。
- 向模型负责人和可视化负责人提供可直接使用的数据。

建议负责目录：

- `data/`
- `src/data/`
- `notebooks/` 中与数据探索、数据清洗相关的文件

阶段性交付：

- 数据字段说明。
- 数据清洗流程。
- 可用于建模和分析的处理后数据。

### 2. 模型负责人

主要负责构建共享单车再平衡相关的预测或优化模型。

具体任务包括：

- 根据项目目标确定模型输入、输出和约束条件。
- 构建需求预测、供需差异分析或车辆调度优化模型。
- 编写模型训练、求解和评估代码。
- 与数据负责人确认模型所需数据格式。
- 向结果分析与可视化负责人提供模型结果。

建议负责目录：

- `src/models/`
- `notebooks/` 中与模型实验相关的文件
- `outputs/` 中与模型输出相关的文件

阶段性交付：

- 基线模型。
- 模型输入输出说明。
- 模型结果与评价指标。

### 3. 结果分析与可视化负责人

主要负责解释数据和模型结果，并通过图表展示项目发现。

具体任务包括：

- 分析不同区域、不同时段的共享单车需求变化。
- 对模型结果进行解释和对比。
- 制作地图、趋势图、热力图、柱状图等可视化结果。
- 总结再平衡前后的效果差异。
- 维护最终可展示的图表和分析结论。

建议负责目录：

- `notebooks/` 中与结果分析、可视化相关的文件
- `outputs/figures/`
- `docs/` 中与结果解释相关的说明

阶段性交付：

- 关键图表。
- 结果分析说明。
- 模型效果对比结论。

## 协作方式

每个人应尽量在自己负责的模块内工作，但需要保持数据、模型和结果之间的接口清晰。

推荐协作流程：

1. 数据负责人先整理基础数据格式。
2. 模型负责人基于处理后的数据建立初始模型。
3. 结果分析与可视化负责人根据数据和模型输出制作分析结果。
4. 三人共同确认模型结果是否合理，并根据分析结果调整数据处理或模型方案。

## Git 使用约定

建议不要直接在 `main` 分支上完成所有修改。每个人可以根据任务创建自己的分支。

分支命名示例：

```text
data/preprocessing
model/rebalancing-baseline
visualization/result-analysis
docs/project-notes
```

提交信息应简洁说明改动内容，例如：

```text
Add data preprocessing script
Implement baseline rebalancing model
Create demand heatmap visualization
Update data field documentation
```

## 文件放置约定

- 原始数据放在 `data/`，大文件不要直接提交到仓库。
- 可复用代码放在 `src/`，不要只散落在 notebook 中。
- 探索性分析可以放在 `notebooks/`。
- 图表、模型输出和实验结果放在 `outputs/`。
- 重要方法说明、数据说明和任务说明放在 `docs/`。

## 当前优先事项

项目初期建议按以下顺序推进：

1. 确认数据来源和字段含义。
2. 完成基础数据清洗。
3. 明确模型目标、输入、输出和评价指标。
4. 建立一个可运行的基线模型。
5. 对需求分布和模型结果进行可视化分析。
6. 根据分析结果迭代数据处理和模型设计。
